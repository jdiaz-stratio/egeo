Design and behaviour

Spinners are used to give feedback to the users about some action it triggers clicking or acting on someplace in the UI.

This type of feedback is mandatory when the user can suffer a potential long-wait process. Keep in mind that server requests always are potential long-wait processes due to lag or connection problems.

With spinners, we avoid user doubts about if the action was triggered and, at the same time, avoid potential clicking retries.

A spinner can be used in different places due to different situations along the app use lifecycle. So, must be flexible and adaptable on its design.

Some sample uses of the same spinners could be:

  • Full-screen when the whole page must be blocked until the process ends.
  • Inside a button when the user triggers a long-wait action clicking on the button.
  • Inside a table row when the user triggers a long-wait action clicking somewhere on the table row.
  • Over a table when the user triggers a long-wait action that blocks the full table.
  • etc.

Usage and best practices

Below are the good practices that should guide the uses of the component to maintain coherence in the design system and the spirit with which they were sustained.

  • Use the spinner always the user can potentially wait for long times waiting (more than 1 second).
  • Combine the use of the spinner with disabled modes or overlay blockers to give the user the right feedback about what items on your UI are blocked due to the action triggered.
  • Control potential connection problems to avoid eternal spinners, giving the user a message about the problem.
  • When an action could take several minutes or more, it must include regular changing messages to give the user the feeling that its process is running. Try always to complete messages with percentage information.
